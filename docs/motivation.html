--- title: "pree: why?" layout: ./_layout.html --- ## Why `pree`? TLDR: üõ†Ô∏è for building websites, we spend too much time wrestling with tools. ‚òï we can go [no build](https://world.hey.com/dhh/once-1-is-entirely-nobuild-for-the-front-end-ce56f6d7) with modern web features ([ESM](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules), [Web Components](https://developer.mozilla.org/en-US/docs/Web/API/Web_components), etc.). üöß there are a few fundamental gaps though (e.g. web components render on client). üß© `pree` fills these gaps as leanly as possible. --- A ton of modern web development is wrestling with build tools. A package has a new module format, your transpiler doesn't like that. The work-around pisses off the bundler. You fix that with a weird trick, but that messes up your test-runner, now your coverage reports are screwed up. You spend hours on getting your build tools to work, without having built anything. There is a chance that you don't need many of these tools anyways. Bundlers were designed to handle dependencies and [optimize asset delivery](https://hpbn.co/optimizing-application-delivery/#optimizing-for-http1x). Nowadays we can handle dependencies perfectly with [ESM](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules) and [import maps](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/script/type/importmap), and with [HTTP/2 multiplexing](https://web.dev/articles/performance-http2#request_and_response_multiplexing), in a lot of cases bundling doesn't optimise anything, sometimes even [lowering the performance](https://hpbn.co/optimizing-application-delivery/#minimize-concatenation-and-image-spriting). Modern CSS with its [nested styles](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_nesting/Using_CSS_nesting), [CSS variables](https://developer.mozilla.org/en-US/docs/Web/CSS/Using_CSS_custom_properties), [CSS layers](https://developer.mozilla.org/en-US/docs/Web/CSS/@layer), etc. has filled the gaps that likes of [Sass](https://sass-lang.com/) and [Less](http://lesscss.org/) were designed to fill. We got [React](https://reactjs.org/), [Angular](https://angular.io/), [and others](https://en.wikipedia.org/wiki/Single-page_application#JavaScript_frameworks), to break our websites into isolated, reusable nuggets of logic, style and layout (i.e. components), but [Web Components](https://developer.mozilla.org/en-US/docs/Web/Web_Components) do provide all of that, alongside [the possibility of server-side rendering](https://developer.chrome.com/docs/css-ui/declarative-shadow-dom), in an interoperable standard manner (e.g. you can use any web component anywhere without having to care about the framework it was built with). So can't we just shed all that weight and build websites using [no build tools](https://world.hey.com/dhh/you-can-t-get-faster-than-no-build-7a44131c) at all? _Well ..._ --- As great as that sounds, in practice, there are still gaps. Even for smaller projects like documentation sites or blogs, which have fewer moving parts and require less logic on the client, there are few, but fundamental, shortcomings: üß¨ Web components render on client by default. This makes user experience janky and slow. You can pre-render them, but that would require a build step.
üèóÔ∏è Websites typically have multiple pages, sharing layout and tons of metadata, differing in a few. Without a build step, managing these shared and differing parts would be a headache.
üëª Many components are static: they don't do anything on the client. Removing their code, you wouldn't just be cutting down on downloaded assets, but also on unnecessary work the client has to do. But then again, that wouldn't be possible without a build step. --- This is where `pree` comes in. It is a little tiny tool that meerly fills those gaps, as simply as it can, without changing much. It is specifically designed for building static websites (websites with fixed pages), as conveniently as possible, while being as "no build" as possible: - It [prebuilds HTML files](usage/index), pre-rendering all web components using [declarative shadow DOM](https://developer.chrome.com/docs/css-ui/declarative-shadow-dom). - It [handles metadata and layout](usage/meta-layout) with front matter. - It [handles build-only components and scripts](usage/build-only-scripts). `pree` basically opens each HTML page as a client would, fetches necessary scripts and components and lets them populate the page, then saves the resulting HTML. Now when the clients get these pages, they are spared this initial wait time, as they get prebuilt HTML with pre-rendered components, and are also spared scripts, components and computation that were only needed for that initial rendering. --- ##### Q: Isn't this just another build tool? **A**: It is, but a really thin and invisible one. With the exception of layouting, what you write is standard HTML/CSS/JS that browsers could execute without any build step, `pree` just makes them faster. Even after building, the client receives the exact same JavaScript, CSS and _almost_ the same HTML that you wrote*. * Except, again, for a bit of layouting, and some pre-rendered web components. ##### Q: What if I want to write Markdown? **A**: You can. Just use a web component like [zero-md](https://zerodevx.github.io/zero-md/). [Load it `build-only`](usage/build-only-scripts.html) so that your users get prebuilt HTML and dont have to download a markdown parser. ```html <script build-only type="module" src="https://esm.sh/zero-md"></script> <zero-md src="my-markdown.md"></zero-md> ``` In fact, [these docs are written](https://github.com/loreanvictor/pree/blob/main/docs/motivation.html) in markdown (with a [custom component](https://github.com/loreanvictor/pree/blob/main/docs/assets/components/m-d.js)) and built with `pree`: ```md <m-d> ## Why `pree`? TLDR: - for building websites, we spend too much time wrestling with toolchains - we can go _no build_ with modern web features (ESM, Web Components, etc.) ... </m-d> ``` ##### Q: What if I want to split HTML files into parts? **A**: Web components can do that too. Web components are pretty powerful, becoming more powerful with `pree` as they can do magical stuff on build time without affecting client-side performance and user experience. For example, this is how splitting is handled in these docs: ```html <script build-only type="module" src="./assets/components/use-html.js"></script> <aside> <use-html src="./_toc.html"></use-html> </aside> ``` ##### Q: What sort of websites can be built with `pree`? **A**: Static websites, like blogs, documentations, landing pages, etc. For dynamic pages (e.g. e-commerce), use another tool. Note that _static_ doesn't mean _non-interactive_, just that you have fixed pages. `pree` [can be used](api.html) for more complex use-cases as well. However, its main responsibility is to wait for an initial page load so users don't have too, which puts a cap on how fast can it prebuild a single page. With a bunch of fixed pages, you can speed this up with parallelisation (which `pree` does), but that might not be feasible in other contexts. ##### Q: What kind of components work with `pree`? **A**: ALL web components work with `pree`. Static ones that are loaded _build-only_ will be faster, as will interactive ones supporting SSR through [declarative shadow DOM](https://developer.chrome.com/docs/css-ui/declarative-shadow-dom). ##### Q: Writing web components is hard though, right? **A**: With proper tools, its a breeze. I use [minicomp](https://github.com/loreanvictor/minicomp), and combine it with [rehtm](https://github.com/loreanvictor/rehtm) for more interactive components. They support SSR and are quite simple. ```js import { define, currentNode } from 'https://esm.sh/minicomp' define('use-html', ({ src }) => { const host = currentNode() fetch(src) .then(res => res.text()) .then(html => { host.innerHTML = html }) return '' }) ``` ##### Q: Can you really build cool stuff without a build tool? **A**: Judge for yourself. These docs are built with `pree` and no other tooling: check the [source code](https://github.com/loreanvictor/pree/tree/main/docs). ##### Q: Whats with the front matter? Why not pure HTML? **A**: I'm salty about this, but I couldn't come close to the developer experience of layouting with front matter using web components. As powerful as they are, web components are about _isolation_, which isn't what you want with layouting.
