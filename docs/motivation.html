--- title: "pree: why?" layout: ./_layout.html --- ### No Time To Build A ton of modern web development is wrestling with build tools. Transpilers, bundlers, runtimes, test runners, devtools, packages with various module formats, frameworks with custom syntax and custom devtools, all need to play nice together. In the end, more time is spent configuring build tools instead of building. But we don't need to deal with this mess anymore. Modern web features such as [ESM](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules), [import maps](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/script/type/importmap), [Web Components](https://developer.mozilla.org/en-US/docs/Web/Web_Components), [nested styles](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_nesting/Using_CSS_nesting), [CSS variables](https://developer.mozilla.org/en-US/docs/Web/CSS/Using_CSS_custom_properties), etc. enable building websites by writing plain HTML, CSS and JavaScript, no build tools involved.
---
Well, thats actually just the idea. In practice, if you have big applications, you still might want to use bundlers (though [it is possible](https://world.hey.com/dhh/once-1-is-entirely-nobuild-for-the-front-end-ce56f6d7) to not build your application in such a way). Even for smaller projects like documentation sites or blogs, which have fewer moving parts and require less logic on the client, there are inherent short comings: üß¨ Web components render on client. This means inevitably jankier user experience. You'd need a build step to pre-render them and get good perceived speed and performance.
üèóÔ∏è Your website typically has multiple pages. They share layout and differ in metadata. You'd need a build step, or you'd need to do tons of copy pasting yourself.
üëª In many cases most components are just stylistic: they don't do anything on the client. You'd need a build step to remove them from being sent to the client, greatly increasing page speed by simply shipping less.

### Here comes `pree` `pree` aims to fill in these gaps and basically just that. It is specifically designed for building static sites such as blogs or documentations. The goal is to leverage web standards to provide the most smooth and simple developer experience for building such websites. - It prebuilds HTML files, pre-rendering all web components using [declarative shadow DOM](https://developer.chrome.com/docs/css-ui/declarative-shadow-dom). - It handles metadata with [front matter](https://www.scribendi.com/academy/articles/front_matter.en.html#:~:text=Front%20matter%20is%20the%20first,a%20preface%2C%20and%20much%20more.), handling layouting with `` elements. - It handles build-only components and scripts, by simply removing `<script build-only>` tags. This means, barring the front matter, you will be writing HTML, CSS and JavaScript that browsers could simply just load. `pree` just makes stuff faster by pre-running some scripts and removing some, resulting in a basically no-build developer experience that also rocks top-tier performance. ‚ú® Ok to be 100% honest, `pree` also does a little thing on top: it exposes special APIs accessible by scripts and components during build time, allowing them to access information about the build environment (such as git data, filesystem information, environment variables, etc.). This allows web components to act even smarter, while remaining standard compliant.
