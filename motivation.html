No Time To Build

A ton of modern web development is wrestling with build tools. Transpilers, bundlers, runtimes, test runners, devtools, packages with various module formats, frameworks with custom syntax and custom devtools, all need to play nice together. In the end, more time is spent configuring build tools instead of building.

Should we keep torturing ourselves though? Modern web features such as ESM, import maps, Web Components, declarative shadow DOM, nested styles, CSS variables, etc. enable building nice smooth experiences by writing plain HTML, CSS and JavaScript, no build tools involved. Perhaps its time to adopt a "no build" approach, embracing the simplicity and speed of the modern web.



In practice, there are still gaps. Big applications still might want to use bundlers, though it is possible to avoid them. Even for smaller projects like documentation sites or blogs, which have fewer moving parts and require less logic on the client, there are fundamental short comings:

Here comes pree

pree aims to fill in these gaps as leanly as it can. It is specifically designed for building static sites such as blogs or documentations. The goal is to provide a simple, straightforward and smooth developer experience for building static websites, while being as "no build" as possible.

  • It prebuilds HTML files, pre-rendering all web components using declarative shadow DOM.
  • It handles metadata with front matter, handling layouting with <slot> elements.
  • It handles build-only components and scripts, by simply removing <script build-only> tags.

This means, barring the front matter, you will be writing HTML, CSS and JavaScript that browsers could simply just load. pree just makes stuff faster by pre-running some scripts and removing some others, resulting in an almost no-build developer experience that also rocks top-tier performance.



Q: Isn't this just another build tool?

A: It is, but a really thin and invisible one. With the exception of layouting, using and not using pree gets you basically the same website. pree just speeds the website up by pre-rendering it.

Q: What if I want to write Markdown?

A: You can. Just use a web components like zero-md. Load it build-only, and pree, and your users get prebuilt HTML and dont have to download a markdown parser.

Q: What if I want to have split an HTML file into parts?

A: Web components can do that for you. Web components are pretty powerful. With pree, they can become even more powerful as they can do magical stuff on build time without affecting client-side performance and user experience.

Q: What sort of websites can be built with pree?

A: Static websites, like blogs, documentations, landing pages, etc. For dynamic pages (e.g. e-commerce), use another tool. Note that static doesn't mean non-interactive, just that you have fixed HTML files.

Q: What kind of components work with pree?

A: ALL web component works with pree. Static ones that are loaded build-only will be faster. Interactive ones supporting SSR through declarative shadow DOM will be much faster.

Q: Can you really build cool stuff without a build tool?

A: Judge for yourself. These docs are built with pree and no other tooling: check the source code.

Q: Whats with the front matter? Why not pure HTML?

A: I'm salty about this, but I couldn't come close to the developer experience of layouting with front matter using web components. As powerful as they are, they are about isolation, which isn't what you want with layouting.
